# Agent Memory — skill bundle for Codex

Three skills that give a Codex agent persistent, per-project memory layered on top of [graphify](https://github.com/safishamsi/graphify):

| Skill | Trigger | What it does |
|---|---|---|
| `remember` | `$remember <thing>` | Stash one fact / decision / lesson / note as structured JSON + markdown mirror |
| `compact`  | `$compact`          | Summarize the current session into a JSON with `summary` + `raw_context` split, then fold into the graph |
| `recall`   | `$recall <topic>`   | Two-tier retrieval: cheap summary-level first; loads `raw_context` only if summary is insufficient or the user replies `dig` |

Memory is **per-project-root** — all files live under the current working directory. Open the same folder in a new Codex session, memory is still there. Open a different folder, memory is empty (by design — projects don't bleed into each other).

## Install

1. **Drop the skill folders into Codex's skill directory:**
   ```bash
   cp -r remember compact recall graphify ~/.agents/skills/
   ```
   (Codex also accepts `~/.codex/skills/` depending on version — check your Codex docs.)

2. **Enable multi-agent if you want `$compact` to run in parallel with other work.** In `~/.codex/config.toml`:
   ```toml
   [features]
   multi_agent = true
   ```

3. **Verify triggers are registered.** Start Codex, type `$` — autocomplete should show `$remember`, `$compact`, `$recall` as top-level skills.

## Dependencies

- **`jq`** — required by `$recall` for JSON filtering.
  ```bash
  brew install jq            # macOS
  apt-get install jq         # Debian/Ubuntu
  ```
- **`graphify` CLI** *(optional)* — when present, `$compact` folds memory records into `graphify-out/graph.json` so code nodes and memory nodes link up; `$recall` queries the merged graph.
  ```bash
  pip install graphifyy      # note the double 'y'
  ```
  If `graphify` isn't installed, the skills still work — records just sit on disk as plain JSON/MD files without being indexed into a graph.

## On-disk layout

Everything lives under the current project root. Nothing goes in `~/` or any shared location.

```
<project>/
├── memory/                                 ← $remember writes here
│   ├── decisions/<ts>-<slug>.{json,md}
│   ├── lessons/<ts>-<slug>.{json,md}
│   ├── facts/<ts>-<slug>.{json,md}
│   └── notes/<ts>-<slug>.{json,md}
├── notes/
│   └── sessions/<ts>-<topic>.{json,md}     ← $compact writes here
└── graphify-out/                           ← generated by `graphify` CLI
    └── graph.json                          ← unified graph (code + memory)
```

Keep `./memory/` and `./notes/` in git if you want memory to travel with the repo. `graphify-out/` is derived and safe to `.gitignore`.

## Storage: separate folders. Search: one unified surface.

Files live in three folders on disk for *organization* reasons (authored vs derived, per-skill scope). At **query time** they collapse into one search surface — the layers never stay separate.

**With graphify installed** — `graphify update .` ingests everything into one graph:

```
               (code)                DigestAuth
                                          │
                           ┌──────────────┼──────────────┐
                           │  linked via  │ linked via   │
                           │  "nodes"     │ "nodes_ref"  │
                           ▼              │              ▼
              (memory)                    │          (session)
          "why DigestAuth"                │         "auth-redesign"
          decision record                 │         session compact
                                          │
                                     (all in one
                                      graph.json)
```

`$recall "DigestAuth"` then walks this unified graph — starting from the code node, traversing edges to the memory records and session nodes that cite it.

**Without graphify** — `$recall` still unifies at search time, just via the filesystem:

```
$recall "DigestAuth"
        │
        ▼
   find memory/ notes/sessions/ -name "*.json"      ← one find, both folders
        │
        ▼
   jq filter across every file                      ← one search pass
        │
        ▼
   results mix decisions + lessons + sessions       ← one ranked answer
```

So: separate on disk, unified at search. The split only matters for `.gitignore` and maintainability; it never shows up in query results.

## Design notes

- **One graph, two kinds of input.** Code is parsed by graphify into the graph. Memory files are ingested by `graphify update .` and linked to the code nodes they name. `$recall` walks one unified graph.
- **Summary + raw_context split.** `$compact` writes `summary` (3–6 bullets) alongside `raw_context` (the substrate). `$recall` loads only `summary` by default; `raw_context` is gated on an explicit `dig` to keep retrieval cheap.
- **No invented nodes.** `$remember` and `$compact` must only cite node labels present in `graphify-out/graph.json` or explicitly named by the user. Free-text labels create orphan nodes and break link-back.
- **Append-only sessions.** `$compact` never overwrites prior session files — multiple compacts per session are expected and preserve distinct phases.

## Trigger syntax

Codex picks up the skill **by top-level skill name**, not by sub-commands inside one skill. That's why this bundle ships three separate skill folders — one per trigger — rather than a single `remember` skill with internal `$compact` / `$recall` branches.
